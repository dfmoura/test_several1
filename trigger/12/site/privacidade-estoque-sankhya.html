Política de Privacidade: Estoque Sankhya

Aplicativo Android (Google Play: Estoque ERP Sankhya Lite)

Última atualização: 2 de junho de 2026 (versão do app 1.0.8)

1. Quem é o responsável

Responsável: TRIGGER DESENVOLVIMENTO PROFISSIONAL LTDA (nome fantasia TRIGGER DATA INTELLIGENCE, CNPJ 53.369.941/0001-63).
Contato: diogo.moura@triggerti.com

2. Objetivo desta política

Esta política descreve quais dados são utilizados no app Estoque Sankhya (nome na loja: Estoque ERP Sankhya Lite) para autenticação e consulta de estoque via API Sankhya. O texto reflete o comportamento implementado no aplicativo (chamadas de rede e armazenamento local de credenciais).

O app encaminha credenciais fornecidas pelo usuário e realiza consultas no servidor indicado na configuração. A semântica das requisições e do endpoint de estoque está na documentação oficial do Sankhya.

3. Como o app funciona

Você informa um código de produto e configura o acesso ao Sankhya (ou ativa o modo demonstração). O app então:

  • Autentica via OAuth2 (POST na URL configurada, em geral https://api.sankhya.com.br/authenticate) com cabeçalho X-Token (token do Gateway Sankhya Om).
  • Usa o access_token retornado (Bearer) para consultar estoque via GET em /v1/estoque/produtos/{codigoProduto}.
  • Opcionalmente, consulta via View customizada (loadView), quando existir no ERP e você escolher essa fonte.
  • No modo demonstração, exibe dados fictícios locais, sem chamada à API do seu ambiente.
  • Quando a build exige licenciamento, valida uma chave de licença com servidor Trigger (ver seção 4.5).

4. Quais dados são utilizados

4.1 Credenciais informadas pelo usuário

URL de autenticação e X-Token (Gateway) podem ser salvos no dispositivo. Client ID e Client Secret OAuth2 do parceiro Trigger vêm embutidos na build; não são digitados na tela. O app não solicita nome, e-mail ou outros dados pessoais além do necessário para API e licença.

4.2 Dados enviados na autenticação

No POST OAuth2, o app envia grant_type=client_credentials, client_id e client_secret no corpo, e o X-Token no cabeçalho HTTP. Não há envio de usuário/senha de operador nesse fluxo.

4.3 Consulta de estoque

A consulta usa Authorization: Bearer {bearerToken} no GET de estoque. Conforme a documentação Sankhya, o endpoint considera estoque registrado no ERP (sem WMS), estoque próprio da empresa, e produtos com movimentação. Detalhes em getestoqueporproduto.

4.4 Dados retornados pela API

Registros de estoque são exibidos na tela durante o uso; não há persistência local desses resultados como parte do fluxo de credenciais.

4.5 Licenciamento (quando habilitado na build)

Para builds com validação ativa, o app envia ao servidor de licenças Trigger (funções hospedadas em infraestrutura Supabase): a chave de licença informada, um identificador estável do aparelho (Android ID, quando disponível) e o identificador do produto (estsankhya). A chave pode ficar armazenada localmente de forma mascarada. O objetivo é vincular a licença ao dispositivo na implantação assistida.

5. Armazenamento local e segurança

URL de autenticação e X-Token ficam em SQLite no dispositivo; dados sensíveis usam cifragem AES com chave no Flutter Secure Storage. Preferências como modo demonstração ficam em tabela local de configurações. Ao salvar novamente, as credenciais anteriores são sobrescritas.

6. Compartilhamento

Dados de estoque e autenticação são enviados aos servidores Sankhya configurados (API oficial do parceiro). Dados de licença, quando aplicável, vão apenas ao servidor de licenças Trigger para ativação e verificação de uso. Não há compartilhamento com terceiros para publicidade.

7. Publicidade e analytics

Com base no projeto atual, não há integrações de anúncios nem analytics de marketing no app. Se isso mudar, esta política será atualizada.

8. Retenção

Credenciais e chave de licença permanecem no dispositivo até serem substituídas, removidas no painel do app ou até o usuário limpar os dados do aplicativo nas configurações do Android.

9. Seus direitos

Para informações sobre tratamento de dados, contate diogo.moura@triggerti.com. Para remover credenciais localmente, use a opção de limpar dados do aplicativo no sistema.

10. Alterações

Podemos atualizar esta política quando o app mudar. A data no topo indica a versão vigente.
